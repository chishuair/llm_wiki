## 一、安装与首次启动

### 1.1 三端安装包

| 平台 | 文件 | 安装方式 |
|---|---|---|
| **macOS** | `案件知识库_x.x.x_aarch64.dmg` | 双击打开 → 将应用拖进「Applications」 |
| **Windows** | `案件知识库_x.x.x_x64-setup.exe` | 双击运行安装程序，按提示点下一步即可 |
| **统信 UOS / Ubuntu / Debian** | `案件知识库_x.x.x_amd64.deb` | 双击打开「软件安装器」；或命令行 `sudo dpkg -i xxx.deb` |
| **任意 Linux** | `案件知识库_x.x.x_amd64.AppImage` | 赋予执行权限 `chmod +x *.AppImage` 后双击 |

### 1.2 安装本地大模型（Ollama）

应用完全依赖**本地大模型**，首次使用前请安装 Ollama：

1. 访问 <https://ollama.com/download>，下载并安装
2. 安装完成后，命令行运行：
   ```bash
   ollama pull qwen2.5:14b
   ```
   推荐模型（任选其一，视机器性能）：
   - **`qwen2.5:14b`**（16GB 内存起）—— 默认推荐
   - **`qwen2.5:32b`**（32GB 内存起）—— 更好的说理质量
   - **`qwen2.5:7b`**（8GB 内存起）—— 小内存应急

3. 让 Ollama 保持运行（默认监听 `http://localhost:11434`）

### 1.3 首次打开应用

- 启动后会看到**欢迎页**：案件知识库标题 + 两个大按钮「新建案件库」「打开案件文件夹」
- 下方是**案件列表**（第一次使用为空）
- 图标栏在最左侧（从上到下）：
  - 💼 切换案件
  - 📄 案件问答
  - 📁 案件材料
  - ⚖️ 法律依据
  - ✨ 法律文书
  - 🔍 全文检索
  - 🕸️ 关联图谱
  - 📋 规范检查
  - 📝 待办审阅
  - 🌐 深度研究（默认关闭，法院内网场景不使用）

### 1.4 法院内网部署检查

首次部署到法院内网电脑时，建议逐项确认：

| 项目 | 要求 | 检查位置 |
|---|---|---|
| **本地/内网大模型** | Ollama 或 OpenAI 兼容内网服务可访问 | 「设置」→ 服务地址 |
| **预置法规包** | 安装包内置 `lawbase-pack-full/lawbase-pack.json`，启动后自动导入 | 「法律依据」页面应显示法规总数 |
| **OCR 引擎** | 推荐安装 PaddleOCR；Tesseract/ocrmypdf 可作为兜底 | 「设置」→ OCR 识别环境 |
| **PDF 解析组件** | PDFium 随安装包提供 | 打开 PDF 材料可预览文字 |
| **数据健康检查** | 检查材料、主数据、法规库、OCR、开庭工作单 | 「辅助工具」→ 数据健康检查 |

离线部署建议：

1. 外网电脑提前准备安装包、PaddleOCR wheel 包、模型文件、Ollama 模型文件、法规包。
2. 通过 U 盘或内网软件源导入法院内网电脑。
3. 启动应用后先进入「设置」检测 OCR，再进入「法律依据」确认法规包已加载。
4. 新建案件后进入「辅助工具」→「数据健康检查」，确认环境和案件数据可用于生成文书。

## 二、案件管理

### 2.1 新建案件库

**操作路径**：欢迎页「+ 新建案件库」 或 应用内左侧 💼 → 「+ 新建案件」。

弹窗里需要填：

- **案件名称**：例如 `合同纠纷案-2026-001`
- **选择模板**：四种案件类型（下一节详解）
- **目录结构预览**：展开看要创建的目录；非必选的目录可以取消勾选
- **案件父目录**：选一个文件夹来保存案件（例如 `~/Desktop/案件库/`）

点**创建案件库**后，应用会自动生成该案件的目录结构。

### 2.2 四种案件模板

| 模板 | 图标 | 适用场景 | 特色目录 |
|---|---|---|---|
| **民事案件** | ⚖️ | 合同纠纷、侵权、婚姻家庭、劳动争议 | 九大核心模块 + 庭审笔录/办案手记/类案检索/文书台账 |
| **刑事案件** | 🛡️ | 普通刑事一审 | 被告人信息、指控犯罪事实、供述与辩解、辩护意见、量刑情节 |
| **行政案件** | 🏛️ | 行政诉讼 | 被诉行政行为、主体适格审查、六维合法性审查 |
| **执行案件** | ⚙️ | 民事执行 | 执行依据、财产调查、强制措施、异议复议、终结情形 |

### 2.3 切换、返回案件

- **左侧图标栏顶部的 💼**：弹出案件下拉，展示所有已创建的案件，点击即切换
- **欢迎页**：点案件卡片直接打开
- **左侧图标栏最底 ⇄**：切换到欢迎页，不关闭现有案件

### 2.4 删除案件

应用里不提供删除按钮（避免误删）。  
要彻底删除一个案件，用**访达/资源管理器**删除对应的案件文件夹即可。

## 三、导入案件材料

### 3.1 支持的文件类型

| 类型 | 文件 | 处理方式 |
|---|---|---|
| **PDF** | `.pdf` | 优先提取文字层；扫描件可用 PaddleOCR 离线识别 |
| **Word** | `.doc / .docx` | docx-rs 提取文字 |
| **Excel** | `.xls / .xlsx` | calamine 提取 |
| **文本** | `.txt / .md / .log` | 直接读取 |
| **图片** | `.png / .jpg 等` | PaddleOCR 离线识别，失败时回退 Tesseract |
| **视频/音频** | `.mp4 / .mp3 等` | 存档 |

> 建议在内网电脑预装 PaddleOCR。应用本身不联网，OCR 模型和 Python 依赖应提前离线部署。

PaddleOCR 安装示例：

```bash
python3 -m pip install paddlepaddle paddleocr
```

在不能联网的法院内网环境，应先在外网机器下载 wheel 包和模型文件，再通过 U 盘/内网软件源安装。

### 3.2 导入操作

1. 左侧图标栏点 **📁 案件材料**
2. 右上角两个按钮：
   - **+ 导入材料**：选单个或多个文件
   - **+ Folder**：整个文件夹批量导入
3. 导入后**自动开始** LLM 分析（后台队列）

### 3.3 查看处理进度

- 左下角 **「队列进度」** 面板：展示正在处理的文件与状态
- 每份材料处理大约耗时 **3-10 分钟**（取决于文件长度和模型速度）
- 处理完成后，左侧「知识目录」会**实时刷新**，出现 LLM 提取的页面

### 3.4 手动重新处理一份材料

如果某份文件的处理结果不满意，或者你更新了 schema：

- 每行材料右边有一个 **✨ 魔法棒图标**
- 悬停提示「让 AI 提取并归类到知识库」
- 点一下即可**重新入队**

### 3.5 材料预览

点击任意材料文件名 → 右侧预览区自动展示**提取后的全文**，可复制引用。

## 四、知识目录与自动分类

### 4.1 九大业务目录（民事模板）

LLM 导入材料后，会自动把信息**分门别类**写入：

| 目录 | 存放什么 |
|---|---|
| **案情概述** | 案件基本事实、时间线、程序摘要 |
| **当事人信息** | 原告、被告、第三人、代理人 |
| **证据清单** | 具体证据及三性审查 |
| **争议焦点** | 审理中需裁判的核心问题 |
| **法院认定事实** | 经审查认定的事实结论 |
| **本院认为** | 说理与法律适用逻辑 |
| **法律依据** | 引用的条款与裁判规则 |
| **判决结果** | 主文、义务、诉讼费用 |
| **审理过程** | 立案、送达、开庭、合议等 |

### 4.2 辅助目录（可选）

- **庭审笔录** — 开庭笔录、质证辩论记录
- **办案手记** — 承办人心证、疑点、合议讨论
- **类案检索** — 指导性案例、参考案例
- **文书台账** — 已发出的各类文书归档

### 4.3 查看与编辑

- **左侧「知识目录」** 按类别折叠展示每个页面
- **「文件目录」** 标签：按文件系统层级查看所有 md
- 点击任一页面 → 右侧打开**编辑器**，可直接修改（自动保存）
- 修改后会自动更新 `wiki/index.md` 索引

### 4.4 自动分类不准？

- 点材料的 **✨ 魔法棒** 重新触发处理
- 或者直接手改页面内容 + 移动到正确目录（用文件管理器）

## 五、证据清单与质证模式

### 5.1 新建结构化证据清单页面

1. 左侧「知识目录」右上角点 **+**
2. 自动创建一份带表格的证据清单页面
3. 右侧弹出**表格编辑器**（而不是普通 markdown）

### 5.2 填写证据

| 列 | 说明 |
|---|---|
| 编号 | 证据1、证据2（也可写「原1」「被1」） |
| 名称 | 证据标题 |
| 提交方 | 原告/被告/第三人/法院调取/其他 |
| 证明目的 | 证明什么 |
| 真实性/合法性/关联性 | 采信/部分采信/不采信/待定 |
| 采信 | 是否最终采信 |
| 原件 | 下拉选择 raw/sources/ 里的原件（自动关联） |

### 5.3 全屏质证模式

1. 证据清单页面右上角点 **「全屏查看」**
2. 进入全屏工作区：
   - **左侧**：完整证据表
   - **右上**：点击某行 → 自动显示该证据关联的 PDF 原件
   - **右下**：三个输入栏（🔵 原告意见 / 🟥 被告意见 / 🟡 本院意见）
3. **键盘快捷键**：
   - `↑ ↓` 切换证据条目
   - `⌘+↵` / `Ctrl+↵` 在意见栏之间跳转
   - `ESC` 退出全屏

开庭当天一套键鼠可以高效完成质证笔录。

### 5.4 数据存储

所有数据存在 md 文件的 YAML frontmatter 里，用 Obsidian、VSCode 等工具打开也能看，不被应用锁定：

```yaml
---
type: evidence-list
evidences:
  - id: 证据1
    name: 购销合同
    submitter: 原告
    admitted: true
    sourcePath: raw/sources/合同.pdf
    opinions:
      plaintiff: 原告主张...
      defendant: 被告认可...
      court: 本院认为...
---
```

## 六、法条库（法律依据）

### 6.1 为什么法官必须手动导入

应用**不内置任何法条**。原因：

- 手打条文存在错字、漏字风险
- 法律经常修订，内置内容容易过期
- 确保每一部法律都是**法官亲自从官方渠道获取并核对**的

### 6.2 导入法律

1. 左侧 ⚖️ **法律依据**
2. 右上角 **「导入法律法规」**
3. 选择一个文件：
   - PDF（如 flk.npc.gov.cn 下载的官方 PDF）
   - Word / docx
   - JSON（高阶用户，已有规范 JSON 的）
4. 应用会**自动提取条文**，再由你**在弹出的预览对话框中确认与修改**：
   - 法律名称、别名、生效日期、版本（可自动从文件名提取，也可手填）
   - 每一条条文的条号、章节、正文（可编辑、可删除、可新增）
   - 右侧可开启「对照原文」看 PDF 原始文本
5. 点 **「确认导入」** → 入库

### 6.3 查看、搜索、复制

- **搜索框**：关键词、条号、法名都可以（如 `违约金`、`第577条`、`民诉法 举证`）
- **卡片右上角**两个按钮：
  - 📋 复制引用：得到 `《民法典》第577条`
  - 📚 复制引用+原文：得到 `《民法典》第577条\n当事人一方不履行...`

### 6.4 删除一部法律

- 展开某部法律 → 右上角「删除」按钮
- 重新导入同名法律会**整体覆盖**

### 6.5 LLM 引用硬约束

**这是整个系统最重要的设计**：

- 每次 LLM 生成内容时，应用会把**你导入的法条库**自动塞进提示词
- LLM **只能引用库里真实存在的法律和条款**
- 如果库里没有可适用的法律 → LLM 会返回标识：  
  `【本地法条库未收录相关法律依据，请先在「法律依据」中导入后重试】`
- 应用检测到该标识会在 UI 弹出**「立即导入」**黄色横栏
- 法官可以一键跳转到法律依据页面补充相关法律，然后再生成

## 七、法律文书生成器

### 7.1 可用模板（10 份内置 + 自定义）

| 分类 | 模板名 | 用途 |
|---|---|---|
| 裁判 | 民事判决书（一审） | 合同、侵权、婚姻等一审 |
| 裁判 | 民事裁定书 | 管辖、保全、中止/终结诉讼 |
| 裁判 | 民事调解书 | 双方合意后出具 |
| 裁判 | 离婚纠纷民事判决书 | 含感情/抚养/财产/债务结构 |
| 裁判 | 执行裁定书 | 冻结/扣划/拍卖/终本 |
| 笔录 | 开庭笔录提纲 | 开庭前的辅助庭审指引 |
| 笔录 | 合议庭评议笔录 | 合议庭内部讨论与决议 |
| 程序 | 送达回证 | 送达法律文书签收 |
| 程序 | 审限延长审批表 | 申请延长审限内部审批 |
| 其他 | 结案报告 | 结案归档用内部报告 |

### 7.2 生成一份文书

1. 左侧 ✨ **法律文书**
2. 点任一模板卡片
3. 配置页填：**案号**、**受诉法院**
4. 系统提示会告诉你：
   - 将参考 N 份原件
   - 哪些结构化目录已齐备 / 哪些还空着
5. 点 **「开始生成」**
6. **步骤 1/N 准备材料**（仅首次执行）：  
   对长原件逐份让 LLM 提炼事实要点（带进度条），结果缓存到 `.llm-wiki/source-summaries/`，下次生成同案件时**秒开**
7. **步骤 2/N 逐节生成**：  
   每一节单独调 LLM，实时流式显示；法官可以**边看边修改**

### 7.3 预览页面的操作

- 每节**头部**显示「AI 草稿」徽标（法官可追溯哪些是 AI 生成）
- 每节内容**直接在 textarea 里编辑**，保存到内存（退出前别忘记导出）
- 顶部工具栏：
  - **重新生成**：整份重新跑一遍
  - **打印 / 存为 PDF**：调起浏览器打印界面，选「存储为 PDF」
  - **导出 Word**：下载符合**最高院裁判文书规范**的 `.docx`（仿宋三号、页边距 3.7/3.5/2.8/2.6cm）

### 7.4 法条自动附注

说理或法律依据章节生成完成后，系统会**自动提取并校验引用**：

- 命中法条库的 ✅ 引用 → 在段末附「——引用依据——」块，列出条文原文
- 未命中 → 顶部黄色横栏「立即导入」提示

### 7.5 编辑 / 自定义模板

- 模板卡片悬停，右上角出现 **✏️ 铅笔**
- 点 → 打开完整编辑器：
  - 修改元信息（名称、分类、正标题、说明）
  - 增删改排序章节
  - 每节三种类型可切换：
    - **静态文本**：直接模板文本，支持 `{{case_number}}` 等变量
    - **案件字段**：从案件知识库某个字段自动拷贝
    - **LLM 生成**：写作要点 prompt，LLM 按要点写
- 编辑**内置模板**时会自动**另存为自定义**（原版保留）
- 编辑**自定义模板**时原地保存
- 也支持**导入 JSON 模板**（给进阶用户）

### 7.6 模板自定义示例

比如你希望每次生成「本院认为」都要按本院思路十二步展开说理：
1. 找到「民事判决书（一审）」点 ✏️
2. 找到「本院认为」章节，改 prompt 为你院的写作指引
3. 保存为 `本院民事判决书（一审）` 自定义版

以后就用自己的那张模板生成。

## 八、案件问答（AI 对话）

### 8.1 基本用法

1. 左侧 📄 **案件问答**
2. 左栏点 **+ 新建会话**
3. 底部输入框输入问题，回车或点发送
4. AI 会基于**当前案件的全部知识库页面 + 法条库**作答

### 8.2 AI 使用规则

- 只基于你导入的材料与法条，**不会自行外部检索**
- 涉及法律条款时**严格引用法条库**，库里没有会明确告知「缺××法律，请先导入」
- 引用页面时用 `[[页面名]]` 双链；引用页面编号用 `[1] [2]`
- 每次回答最末尾有隐藏的 `<!-- cited: 1, 3 -->` 注释，记录使用了哪些页面

### 8.3 保存有价值的回答

- AI 每条消息下有 **「保存到知识库」** 按钮
- 保存后会**自动落地**到 `wiki/本院认为/` 目录，命名 `YYYYMMDD-标题.md`
- 下次需要找这段说理，左侧「本院认为」分组下就能看到

### 8.4 重新生成 / 删除会话

- 每条 AI 消息有 **🔄 重新生成**
- 左栏每个会话右键可删除

## 九、全文检索 / 关联图谱 / 规范检查 / 待办审阅

### 9.1 全文检索（🔍）

- 跨全部 md 页面的全文搜索
- 支持关键词、同时命中多个关键词
- 结果展示命中行的上下文

### 9.2 关联图谱（🕸️）

- 把所有知识页面按 `[[双链]]` 可视化成图
- 节点按类别着色（案情概述、当事人、证据……）
- 可交互缩放，找到孤立或中心节点
- 对于复杂案件（多当事人、多证据）非常有帮助

### 9.3 规范检查（📋）

检查知识库的结构化健康度：

- **孤立页面**：没有被任何地方链接
- **失效链接**：双链目标页面不存在
- **无出链**：页面里没有任何引用
- **内容矛盾**（需 LLM）：不同页面里的事实陈述相互冲突
- **内容过时**（需 LLM）：某页面信息与最新材料不一致

### 9.4 待办审阅（📝）

LLM 在 ingest 过程中产出的**需要人工确认的待办**都放这里：

- **缺失页面**：某个重要概念被引用但没页面
- **矛盾**：不同页面存在事实冲突
- **建议**：拓展性提示，如「建议检索类案」
- **重复**：疑似重复的主体

每条待办下面有**操作按钮**：
- **创建页面** / **深度研究** / **跳过** 等
- 解决后自动归档

### 9.5 深度研究（🌐，法院内网慎用）

面向外网检索的补充能力。法院内网不通外网时**不建议使用**；设置里关闭即可。

### 10.1 大模型设置

- **接口地址**：

  - 可改为远程模型接口（如私网部署的 OpenAI 兼容服务）

## 十一、文件组织与数据位置

### 11.1 一个典型案件目录长这样

```
合同纠纷案-2026-001/
├── schema.md               # 目录规范（模板自动生成）
├── purpose.md              # 建库目标
├── raw/                    # 原件区
│   ├── assets/            # 附件（图片、截图等）
│   └── sources/           # 原件（PDF、Word 等）
│       └── .cache/        # 原件提取文本缓存（应用自动管理）
├── wiki/                   # 知识区
│   ├── index.md           # 索引
│   ├── log.md             # 办案日志
│   ├── overview.md        # 案件总览
│   ├── 案情概述/
│   ├── 当事人信息/
│   ├── 证据清单/
│   ├── 争议焦点/
│   ├── 法院认定事实/
│   ├── 本院认为/
│   ├── 法律依据/
│   ├── 判决结果/
│   ├── 审理过程/
│   └── （其他按模板生成的目录）
└── .llm-wiki/              # 应用内部状态（无需手动管理）
    ├── conversations.json  # 会话元信息
    ├── chats/              # 历史对话
    ├── review.json         # 待办审阅
    ├── ingest-queue.json   # 导入队列
    ├── ingest-cache.json   # 导入缓存
    └── source-summaries/   # 长原件事实提要缓存
```

### 11.2 应用级数据（全局，不属于某个案件）

- **法条库**：存在应用的 `app-state.json` 中，跨案件共享
- **自定义文书模板**：同上
- **最近打开案件列表**、LLM 配置：同上

应用级数据位置（通常）：

- macOS: `~/Library/Application Support/案件知识库/`
- Windows: `%APPDATA%/案件知识库/`
- Linux: `~/.config/案件知识库/`

### 11.3 备份建议

**最重要**的是你案件文件夹本身（含 raw/ 和 wiki/）。定期拷贝到外部硬盘或内部存储即可。

应用级数据（`~/Library/Application Support/案件知识库/`）也可以一并备份，恢复时连带法条库、模板都回来。

## 十二、常见问题与排错

### Q1. 上传材料后左侧目录没变化

**可能原因**：
1. LLM 处理时间较长，请等待（左下角「队列进度」）
2. Ollama 没启动或模型没下载

**排查**：
- 命令行 `ollama ps` 看是否有进程
- `ollama list` 看是否有你设置的模型

### Q2. 打开应用弹出「已损坏，无法打开」（仅 macOS）

**原因**：应用未经过 Apple 公证（或公证未完成）。

**临时解决**（在终端执行）：
```bash
sudo xattr -cr "/Applications/案件知识库.app"
sudo codesign --force --deep --sign - "/Applications/案件知识库.app"
open "/Applications/案件知识库.app"
```

### Q3. 生成文书时「本院认为」栏里写着『本地法条库未收录...』

**原因**：LLM 要引用某法条，但你的法条库没导入这部法律。

**解决**：点横栏上的「立即导入」 → 在「法律依据」里导入对应 PDF → 重新生成。

### Q4. 导入法律的 PDF 解析结果乱

**可能原因**：
- PDF 是扫描件（没文字层）→ 优先安装 PaddleOCR 后重新预处理
- PDF 格式非常不规范 → 在预览对话框手动修正

### Q5. 想换本地模型

1. `ollama pull 新模型名`
2. 应用「设置」→ 改模型名 → 保存

### Q6. 案件文件夹移动到别的位置后应用打不开

**解决**：欢迎页 → 「打开案件文件夹」→ 定位到新位置。

### Q7. 如何彻底删除一个案件

1. 应用里切换到另一个案件或欢迎页（不要保持打开被删案件）
2. 访达/资源管理器里**删除案件文件夹**
3. 应用欢迎页的案件列表里该案件会显示加载失败，悬停可以**从列表中移除**（只是清列表，不影响其他）

### Q8. 想和同事分享一个案件知识库

**合规前提**：确认不违反保密规定、当事人隐私。

1. 访达里把整个案件文件夹压缩成 zip
2. 传给同事
3. 同事在应用里「打开案件文件夹」→ 选解压后的文件夹

法条库、自定义模板是全局级别，不会随案件文件夹跟着走。如需同步：

1. 把 `~/Library/Application Support/案件知识库/app-state.json` 拷给对方（覆盖同路径文件需重启应用）

### Q9. 想把某个案件的数据从应用里完全清理（包括缓存）

`rm -rf .llm-wiki/` 即可。不会影响原始材料和知识页面。下次打开会按新状态重建。

## 十三、工作流最佳实践

### 13.1 理想办案流程（以民事案件为例）

1. **立案** → 新建案件（选民事模板）→ 填案号和父目录
2. **收材料** → 起诉状、答辩状、证据扫描件 → 全部扔进「案件材料」
3. **LLM 自动处理** → 10-30 分钟 → 左侧目录自动冒出
4. **核对与补充**：
   - 打开「证据清单」，用表格编辑器整理
   - 打开「当事人信息」，核对姓名、身份证号
   - 打开「案情概述」，补充/修正事实
5. **写案件争议焦点** → 进「争议焦点」新建或由 LLM 提取
6. **开庭前**：
   - 「法律文书」→ 选「开庭笔录提纲」生成
   - 打印 PDF 随庭
7. **开庭中**：
   - 证据清单页面的「全屏查看」进入质证模式
   - 原/被/院意见边问边录入
8. **合议后**：
   - 「法律文书」→ 选「合议庭评议笔录」生成
9. **起草判决书**：
   - 「法律文书」→「民事判决书（一审）」
   - 逐节审阅 AI 草稿、修改
   - 导出 Word，发校对
10. **结案**：
    - 「法律文书」→「结案报告」生成
    - 归档案件文件夹到法院共享盘

### 13.2 法条库建议

- **第一次使用时**：花半天时间，把本院审判常用的 8-12 部法律与司法解释从 flk.npc.gov.cn 下载下来，全部导入
  - 民法典
  - 民诉法
  - 刑法（如办刑案）
  - 刑诉法
  - 行政诉讼法
  - 核心司法解释（民诉法司法解释、合同编解释、证据规定等）
- **日常维护**：关注司法解释更新，有新版本就重新导入覆盖

### 13.3 材料导入建议

- **不要把整个法院共享盘往里灌**，只导入本案**实际证据和材料**
- 卷宗目录、送达回证、通知书等程序文件可以导入但会占 LLM 上下文，建议按「是否对事实认定 / 说理有用」过滤
- 导入后觉得处理不好，先看处理结果再决定是否重跑（右侧的 ✨ 魔法棒）

### 13.4 判决书写作建议

- **LLM 只是草稿工具**，所有事实、数据、引用**必须人工复核**
- 对「本院认为」一节**尤其谨慎**，三段论逻辑、法条引用、涵摄结论三项都要自己判断
- 利用「法条引用校验」栏（md 页面顶部）快速确认所有引用都命中法条库
- 有疑问的事实去 `raw/sources/` 找原件核对（预览面板可以直接打开）

### 13.5 保密与合规

- 应用**完全离线**，不向任何外网发送数据
- 但请确保：
  - 你的电脑本身符合法院涉密电脑管理规范
  - Ollama 本地运行，不要配置远程 API
  - 案件文件夹不要放云同步目录（如 iCloud、OneDrive、Dropbox）
  - 备份到法院内部存储而非公开云

### 13.6 团队协作

目前应用是**单人离线模式**。多人协作的替代方案：

- 一个案件只由一位承办人操作
- 合议庭成员查看：把案件文件夹复制出副本给对方
- 本院标准模板与法条库：由指定管理员维护，定期同步 `app-state.json`
